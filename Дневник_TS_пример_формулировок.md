## Сессия 1 (база типизации)

**Дата и время:** 20.04.2026, вечер  
**Длительность:** ~35 минут

### Шаг 1. Цель сессии
Понять базовые типы в TS и перестать автоматически тянуться к `any`.

### Шаг 2. Изучение темы
Сначала прошёлся по примитивам и посмотрел, где я чаще всего ленюсь с типами.  
Отдельно сравнил `any` и `unknown`, раньше у меня это было почти одно и то же в голове.  
Сейчас понял разницу: `any` отключает проверку, `unknown` заставляет сначала уточнить тип.  
Понравилась мысль про `never`: это не "пусто", а "невозможное состояние".  
Также увидел, что без `strict` я себе же усложняю жизнь, потому что ошибки всплывают поздно.  
В итоге для себя зафиксировал: лучше чуть дольше писать типы сразу, чем потом чинить поведение.

### Шаг 3. Проверка понимания
**5 вопросов:**  
1. Когда допустимо использовать `any`?  
2. Чем `unknown` лучше `any` в повседневном коде?  
3. В каком месте может появляться `never`?  
4. Что даёт `strict` режим на практике?  
5. Почему `null`/`undefined` важно явно учитывать?

**Мои ответы (кратко):**  
1. Временный мост, когда ещё нет ясного контракта данных.  
2. Он не даёт случайно использовать значение без проверки.  
3. Например, в `switch` с полной проверкой вариантов.  
4. Ловит дыры до запуска, особенно на границах данных.  
5. Иначе легко получить падение в рантайме.

### Шаг 4. Разбор ошибки
Сложный момент был про `never`, я сначала объяснил его как "пустой тип".  
Исправил формулировку: `never` значит, что до этого места исполнения дойти нельзя или значение не может существовать.

### Шаг 5. Проверка результата
**Критерии:**  
1. Могу объяснить `any` vs `unknown` на бытовом примере.  
2. Не использую `any` в учебных примерах без причины.  
3. Понимаю, где в коде встречается `never`.

**Статус:**  
1. Выполнено  
2. Частично  
3. Выполнено

### Шаг 6. Следующий шаг
Перейти к union типам и научиться сужать их без `as`.

## Сессия 2 (union, type guards, функции)

**Дата и время:** 21.04.2026, днём  
**Длительность:** ~45 минут

### Шаг 1. Цель сессии
Научиться уверенно сужать union-типы и правильно типизировать функции с ветвлениями.

### Шаг 2. Изучение темы
В этой сессии упёрся в практику: делал несколько маленьких функций с входом вида `string | number | null`.  
Главный инсайт: type guard нужен не "для красоты", а чтобы компилятор мог доказать безопасность ветки.  
Потренировал `typeof`, `in` и проверку через пользовательский guard.  
Понял, почему `as` быстро превращается в костыль: код вроде "зелёный", но по факту риск остаётся.  
Когда начал явно проверять вход, стал лучше видеть слабые места в логике самой функции.  
По ощущениям, качество кода растёт вместе с качеством проверок.

### Шаг 3. Проверка понимания
**5 вопросов:**  
1. Чем сужение через `typeof` отличается от `in`?  
2. Когда нужен user-defined type guard?  
3. Почему `as` не решает проблему, а прячет её?  
4. Как типизировать функцию, которая возвращает разные формы данных?  
5. Где в проекте у меня чаще всего нужна проверка входа?

**Мои ответы:**  
1. `typeof` для примитивов, `in` для проверки свойств объекта.  
2. Когда стандартных проверок недостаточно и нужен повторно используемый предикат.  
3. Потому что это утверждение без доказательства.  
4. Через union на выходе плюс явные проверки по ходу.  
5. На входе в функции обработки заданий и результатов.

### Шаг 4. Разбор ошибки
Я попытался типизировать одну функцию через перегрузки, но сделал это сложнее, чем нужно.  
Правильнее оказалось: сначала union + guard, и только потом думать о перегрузках.

### Шаг 5. Проверка результата
**Критерии:**  
1. Пишу функции без `as` там, где можно обойтись guard.  
2. Могу объяснить, почему компилятор принимает каждую ветку.  
3. Умею выделить проверку в отдельный guard.

**Статус:**  
1. Выполнено  
2. Выполнено  
3. Частично

### Шаг 6. Следующий шаг
Разобрать generics и понять, где они реально упрощают код, а где только усложняют.

## Сессия 3 (generics + применение в проекте)

**Дата и время:** 22.04.2026, вечер  
**Длительность:** ~55 минут

### Шаг 1. Цель сессии
Подтянуть generics и применить типизацию точечно в текущем проекте, а не в абстрактных примерах.

### Шаг 2. Изучение темы
Сделал фокус на своём коде, чтобы не зависнуть в теории.  
Проверил, где можно улучшить контракты данных между компонентами и функциями оценки.  
Generics стали понятнее, когда я начал думать о них как о "шаблоне контракта", а не как о магии с `<T>`.  
Полезно было мысленно задавать вопрос: "что именно должно оставаться неизменным у типа?".  
Ещё заметил, что иногда проще дать явный конкретный тип, чем везде пытаться натянуть generic.  
В итоге выбрал принцип: generic только там, где реально есть повторяемость формы.

### Шаг 3. Проверка понимания
**5 вопросов:**  
1. Когда `interface`, а когда `type`?  
2. В чём практический смысл generic-ограничений (`extends`)?  
3. Когда лучше конкретный тип вместо generic?  
4. Какие utility types уже можно применять без боли?  
5. Где в проекте типизация даст самый быстрый эффект?

**Мои ответы:**  
1. `interface` для расширяемых структур, `type` для композиции и union.  
2. Они защищают от слишком "широкого" входа и улучшают подсказки IDE.  
3. Когда сценарий единичный и переиспользования нет.  
4. `Pick`, `Omit`, `Partial`, `Record`.  
5. На границе данных и в функциях, где есть ветвление логики.

### Шаг 4. Разбор ошибки
Сначала сделал generic "на будущее", но он усложнил чтение функции.  
Откатил на явный тип и стало проще поддерживать. Вывод: не переусложнять ради красоты.

### Шаг 5. Проверка результата

2. Понимаю, где уместен `extends`.  
3. Осознанно выбираю между `type` и `interface`.

**Статус:**

## Карточка знания 1

**Вопрос:** Почему `unknown` безопаснее, чем `any`?  
**Ответ:** `unknown` заставляет сначала проверить тип значения перед использованием, а `any` отключает типовую проверку и может скрыть ошибку до рантайма.  
**Пример:** Когда приходит ответ от внешнего API, сначала держу его как `unknown`, потом валидирую и только после этого работаю с полями.

## Карточка знания 2

**Вопрос:** Когда стоит применять generics?  
**Ответ:** Когда есть повторяемый шаблон логики для разных типов данных, и этот шаблон должен оставаться типобезопасным без дублирования кода.  
**Пример:** Функция-обёртка для обработки результатов, где формат структуры общий, но конкретный payload разный.

# План разработки: портал подготовки к frontend-собеседованию

## 1) Цель проекта
Сделать образовательный портал, где теория и практика идут вместе:
- пользователь читает материал;
- сразу в тексте выполняет интерактивные задания;
- может пройти отдельный тест-режим с итоговым результатом.

---

## 2) Зафиксированные решения (принято)

### MVP для сдачи
- `3` статьи с inline-интерактивом.
- `10–12` заданий в test-режиме с итоговым подсчётом.
- `5` базовых типов заданий:
  - `MultipleChoice`
  - `MatchPairs`
  - `FillTheBlank`
  - `TrueFalse`
  - `OrderSteps`
- `1` кастомный компонент: `Crossword`.
- Все типы работают в двух режимах:
  - `inline` (автономно внутри статьи);
  - `test` (в составе общего теста и прогресса).

### Технологический стек
- `Next.js`
- `TypeScript`
- `MDX` для статей
- локальные данные (без обязательного бэкенда на MVP)
- стили: решение гибкое (можно выбрать удобный подход по ходу разработки)

## 5) Следующий блок обсуждения: архитектура компонентов и данных

### Цель блока
Определить единый контракт для всех типов заданий, чтобы:
- легко добавлять новые компоненты;
- не дублировать логику между `inline` и `test`;
- единообразно считать результаты.

### Вопросы, которые нужно закрыть
1. Какая общая структура данных у любого задания (`id`, `type`, `question`, `explanation`, `difficulty`, `tags`)?
2. Где хранится “правильность” и как не смешивать контент с UI-логикой?
3. Какой единый формат результата проверки (`score`, `maxScore`, `isCorrect`, `feedback`)?
4. Как компонент сообщает результат в test-режиме (`onAnswer`, `onSubmit`, `onStateChange`)?
5. Как один и тот же компонент переключается между `inline` и `test` через `mode`?
6. Где живёт состояние:
   - локально в компоненте;
   - в общем контексте теста;
   - гибридно?

### Рабочая гипотеза (рекомендуемая)
- Общий `ExerciseRenderer`, который по `type` выбирает нужный компонент.
- У каждого типа есть:
  - свой `Prompt` (визуал + ввод),
  - свой `evaluate(answer, exercise)` (чистая функция проверки).
- В `inline` режиме компонент работает автономно.
- В `test` режиме результат уходит в общий `TestSessionContext`.

## 7) Архитектура данных (v1, зафиксировано)

```ts
type ExerciseMode = "inline" | "test";

type ExerciseType =
  | "multiple-choice"
  | "match-pairs"
  | "fill-the-blank"
  | "true-false"
  | "order-steps"
  | "crossword";

interface ExerciseBase {
  id: string;
  type: ExerciseType;
  title?: string;
  question: string;
  explanation: string;
  difficulty?: "easy" | "medium" | "hard";
  tags?: string[];
}
```

Следующий шаг: утвердить `payload` для каждого типа задания и затем объединить их в `Exercise = ExerciseBase & { payload: ... }`.

## 8) Payload моделей (v1, зафиксировано)

```ts
interface MultipleChoicePayload {
  options: { id: string; text: string }[];
  correctOptionIds: string[];
  allowMultiple?: boolean;
}

interface MatchPairsPayload {
  left: { id: string; text: string }[];
  right: { id: string; text: string }[];
  correctPairs: { leftId: string; rightId: string }[];
}

interface FillTheBlankPayload {
  template: string; // "Event loop handles {{blank1}} first"
  blanks: { id: string; answers: string[]; caseSensitive?: boolean }[];
}

interface TrueFalsePayload {
  statement: string;
  correct: boolean;
}

interface OrderStepsPayload {
  steps: { id: string; text: string }[];
  correctOrder: string[]; // step ids
}

interface CrosswordPayload {
  size: { rows: number; cols: number };
  words: {
    id: string;
    clue: string;
    answer: string;
    start: { row: number; col: number };
    direction: "across" | "down";
  }[];
}
```

Рекомендуемое объединение:

```ts
type Exercise =
  | (ExerciseBase & { type: "multiple-choice"; payload: MultipleChoicePayload })
  | (ExerciseBase & { type: "match-pairs"; payload: MatchPairsPayload })
  | (ExerciseBase & { type: "fill-the-blank"; payload: FillTheBlankPayload })
  | (ExerciseBase & { type: "true-false"; payload: TrueFalsePayload })
  | (ExerciseBase & { type: "order-steps"; payload: OrderStepsPayload })
  | (ExerciseBase & { type: "crossword"; payload: CrosswordPayload });
```

## 9) Контракт оценки (v1, зафиксировано)

```ts
interface EvaluationResult {
  isCorrect: boolean;
  score: number; // набрано
  maxScore: number; // максимум
  feedback: string; // короткий фидбек
  details?: {
    correctItems?: number;
    totalItems?: number;
    mistakes?: string[];
  };
}

type EvaluateFn<TExercise, TAnswer> = (
  exercise: TExercise,
  answer: TAnswer
) => EvaluationResult;
```

Правила для MVP:
1. `score` всегда в диапазоне `0..maxScore`.
2. Частичная правильность: `score = maxScore * (correctItems / totalItems)`.
3. `isCorrect = score === maxScore`.
4. В `inline` режиме показываем `feedback + explanation`.
5. В `test` режиме сохраняем `EvaluationResult` и считаем общий итог по сумме `score/maxScore`.

## 10) API компонентов для режимов inline/test (v1, зафиксировано)

```ts
interface ExerciseComponentProps<TExercise, TAnswer> {
  exercise: TExercise;
  mode: "inline" | "test";
  initialAnswer?: TAnswer;
  readonly?: boolean;
  onSubmit?: (answer: TAnswer, result: EvaluationResult) => void;
}
```

Почему так:
- `mode` устраняет дублирование, один компонент работает в двух сценариях.
- `onSubmit` дает единый способ передать результат наружу (особенно для test-режима).
- `initialAnswer` нужен для восстановления прогресса.
- `readonly` пригодится для экрана разбора после завершения теста.

## 11) Контур test-режима (v1, зафиксировано)

```ts
interface TestSessionItemResult {
  exerciseId: string;
  answer: unknown;
  result: EvaluationResult;
}

interface TestSessionState {
  results: Record<string, TestSessionItemResult>;
  totalScore: number;
  totalMaxScore: number;
  answeredCount: number;
  totalCount: number;
}
```

Почему так:
- `results` по `exerciseId` дает O(1)-доступ и простое обновление ответов.
- агрегаты (`totalScore`, `totalMaxScore`) сразу готовы для прогресса и финального экрана.
- хранение `answer` помогает показать подробный разбор без повторного ввода.

## 12) Архитектурный паттерн рендера (v1, зафиксировано)

- `ExerciseRenderer` принимает `Exercise` и `mode`.
- Внутри по `exercise.type` выбирается конкретный UI-компонент.
- Каждому типу соответствует чистая функция `evaluate`.
- UI-компонент отвечает только за ввод и отображение, проверка живет в `evaluate`.

Почему так:
- изоляция логики проверки упрощает тестирование и повторное использование.
- добавление нового типа сводится к двум сущностям: новый UI + новый `evaluate`.
- одинаковое поведение в `inline/test` достигается без копирования бизнес-логики.

## 13) UX-состояния упражнений (v1, зафиксировано)

Состояния:
1. `idle` — ответ еще не отправлен.
2. `answered-partial` — есть частично верный результат.
3. `answered-correct` — ответ полностью верный.
4. `answered-wrong` — ответ неверный.

Базовое поведение:
1. До ответа: кнопка `Проверить`.
2. После ответа в `inline`: `Показать объяснение` и (при наличии лимита попыток) `Попробовать еще`.
3. После ответа в `test`: переход к следующему заданию без показа правильного ответа до финала.

Почему так:
- единая модель состояний подходит всем типам заданий;
- `partial` нужен для честной обратной связи по составным ответам;
- разделение поведения `inline/test` сохраняет обучающий режим и объективность теста.

## 14) Правила попыток и показа ответов (v1, зафиксировано)

1. В `inline` режиме: до `2` попыток на задание.
2. В `inline` после 1-й ошибки: показываем подсказку (короткий `feedback`), но не полный ответ.
3. В `inline` после 2-й ошибки: показываем `explanation` и правильный ответ.
4. В `test` режиме: `1` попытка на задание, без показа правильного ответа до финальной сводки.
5. На финальном экране `test`: показываем разбор по каждому вопросу (ответ пользователя, правильный ответ, объяснение).

Почему так:
- `inline` остается обучающим и не фрустрирует пользователя;
- `test` остается сопоставимым и прозрачным по оценке;
- итоговый разбор дает ценность даже при ошибках и закрывает критерий UX.

## 15) Структура контента: 3 статьи (v1, зафиксировано)

### Статья 1: JavaScript Core для собеседования
- Темы: `Execution Context`, `Call Stack`, `Event Loop`, `microtask/macrotask`, `closures`.
- Inline-упражнения (`3`):
  1. `OrderSteps` по Event Loop.
  2. `FillTheBlank` по lexical scope/closure.
  3. `TrueFalse` по call stack и очередям задач.

Почему так:
- это самые частые и “ловушечные” темы на интервью;
- `OrderSteps` лучше всего проверяет понимание последовательности исполнения.

### Статья 2: React + TypeScript основы
- Темы: `rendering`, `useState/useEffect/useRef`, `controlled/uncontrolled`, `generics`, `narrowing`.
- Inline-упражнения (`3`):
  1. `MultipleChoice` по ререндерам и хукам.
  2. `MatchPairs` (TS utility type -> кейс использования).
  3. `FillTheBlank` по типизации props/events.

Почему так:
- сочетает практику React и TS в формате, похожем на техническое интервью;
- `MatchPairs` хорошо работает на “термин -> применение”.

### Статья 3: Browser APIs + интервью мышление
- Темы: `DOM events`, `Fetch/HTTP`, `Storage`, `Cookies`, `CORS`, объяснение решений.
- Inline-упражнения (`4`):
  1. `TrueFalse` по CORS/cookies.
  2. `MultipleChoice` по HTTP/status codes.
  3. `OrderSteps` (поток запроса и обработка ошибки).
  4. `Crossword` по ключевым терминам (кастомный компонент).

Почему так:
- закрывает browser-блок и “soft technical skill” объяснения;
- `Crossword` добавляет креативность без потери учебной ценности.

Итого по статьям: `10` inline-упражнений.

### Шаблон структуры каждой статьи
1. Короткий ввод: зачем тема важна на интервью.
2. Теоретический блок 1.
3. Inline-упражнение 1.
4. Теоретический блок 2.
5. Inline-упражнение 2.
6. Теоретический блок 3.
7. Inline-упражнение 3 (для статьи 3 — +4-е).
8. Мини-итог: что уметь проговорить вслух интервьюеру.

Почему так:
- интерактив встроен органично и не превращается в “текст отдельно, тест отдельно”;
- сохраняется ритм “узнал -> применил -> закрепил”.

## 16) Структура test-режима: 10–12 заданий (v1, зафиксировано)

Распределение типов:
1. `MultipleChoice` — `3`
2. `TrueFalse` — `2`
3. `FillTheBlank` — `2`
4. `MatchPairs` — `2`
5. `OrderSteps` — `2`
6. `Crossword` — `1` (опционально в тесте, обязательно присутствует в статьях)

Итого:
- базовый вариант: `11` заданий без `Crossword`;
- расширенный вариант: `12` заданий с `Crossword`.

Почему так:
- покрываются разные форматы проверки (факты, последовательность, сопоставление, практическая точность);
- сохраняется баланс сложности и времени прохождения;
- в явном виде выполняется критерий “разнообразие компонентов”.

## 17) Финальный экран test-режима (v1, зафиксировано)

Состав экрана:
1. Итоговый процент и уровень:
   - `0–49%` -> `Need more practice`
   - `50–74%` -> `Almost ready`
   - `75–100%` -> `Junior-ready`
2. Сводка баллов: `totalScore / totalMaxScore`.
3. Прогресс по типам заданий (например: `MultipleChoice 2/3`, `OrderSteps 1/2`).
4. Разбор по каждому вопросу:
   - ответ пользователя;
   - правильный ответ;
   - `feedback` и `explanation`.
5. Кнопки:
   - `Пройти заново`
   - `Перейти к статьям`

Почему так:
- пользователь видит не только “оценку”, но и конкретные пробелы;
- разбор повышает учебную ценность теста и снижает “формальность” проверки;
- кнопки закрывают оба сценария после теста: повтор или дообучение.

## 18) Формат рекомендаций по темам (v1, зафиксировано)

Источник рекомендаций:
- используем `tags` у упражнений (например: `event-loop`, `closures`, `react-hooks`, `cors`).

Правило расчета:
1. Для каждого тега считаем:
   - `tagScore`
   - `tagMaxScore`
   - `tagPercent = tagScore / tagMaxScore * 100`
2. Если `tagPercent < 60`, тег попадает в “зоны для повторения”.
3. В рекомендации показываем топ-3 самых слабых тега.

Формат текста рекомендации:
- `Повторить: Event Loop (40%), CORS (50%), React Hooks (55%).`
- `Рекомендуем прочитать: [статья X], [статья Y].`

Почему так:
- `tags` уже есть в модели данных, не нужна отдельная сложная аналитика;
- порог `60%` простой и понятный для MVP;
- топ-3 ограничивает объем и фокусирует пользователя на главном.

## 24) Минимальный план тестирования (v1, зафиксировано)

Покрытие `evaluate` функций:
1. `multiple-choice.evaluate`:
   - одиночный и множественный выбор;
   - полный/частичный/нулевой результат.
2. `match-pairs.evaluate`:
   - полное совпадение;
   - частичное совпадение;
   - пустой ответ.
3. `fill-the-blank.evaluate`:
   - case-sensitive / case-insensitive;
   - несколько допустимых ответов.
4. `true-false.evaluate`:
   - верный/неверный ответ.
5. `order-steps.evaluate`:
   - правильный порядок;
   - частично правильный порядок.
6. `crossword.evaluate`:
   - заполнено полностью верно;
   - частично верно;
   - есть ошибки.

Smoke-тест сценария:
1. пройти тест от старта до результатов;
2. проверить подсчет `score/maxScore`;
3. проверить блок рекомендаций по `tags`.

Почему так:
- наибольший риск ошибок именно в логике оценки;
- smoke-тест гарантирует, что основной пользовательский путь не сломан.
